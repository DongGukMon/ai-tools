Selective Stage/Unstage — Interaction Model

How should hunk & line-level operations work in the diff view?

A

SourceTree Style — Checkbox Gutter

Each line gets a checkbox in the gutter. Hunk headers get Stage/Unstage buttons. Shift+click for range select. Space toggles selection.

Pros

  • Familiar to SourceTree users
  • Visual feedback per line (checked = will be staged)
  • Multi-select is intuitive

Cons

  • Gutter gets visually busy with checkboxes
  • Checkboxes on every line is heavy for large diffs
B

VS Code Style — Gutter Hover Actions

Hunk header has inline Stage/Unstage/Discard icons. Lines are selected by clicking the gutter (highlight, no checkbox). Right-click or toolbar button to apply action on selection.

Pros

  • Clean gutter — actions only appear on hover
  • Familiar to VS Code users
  • Less visual noise

Cons

  • Hover-only actions are less discoverable
  • Right-click context menu adds complexity
C

Keyboard-First Hybrid

Hunk header buttons (always visible). Line selection via gutter click (highlight band, no checkboxes). Shift+click for range. Space to stage/unstage, j/k to navigate, h for hunk-level action. Focused line has distinct highlight.

Pros

  • Keyboard-driven (matches issue requirement)
  • Clean gutter (no checkboxes)
  • Hunk buttons always visible = discoverable
  • Best of both: mouse click + keyboard nav

Cons

  • Keyboard shortcuts need learning

Recommendation: C (Keyboard-First Hybrid)

Issue explicitly asks for keyboard-first (Space key, SourceTree-level UX). Option C gives the cleanest visual with the strongest keyboard story. Hunk buttons stay always-visible for discoverability, line selection uses gutter highlights (not checkboxes) to keep the diff clean, and j/k/Space gives the fast flow SourceTree users expect.
